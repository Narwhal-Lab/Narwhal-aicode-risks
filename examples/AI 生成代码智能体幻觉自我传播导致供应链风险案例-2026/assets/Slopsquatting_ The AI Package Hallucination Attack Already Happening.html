什么是斜坡蹲？人工智能包幻觉攻击已在发生

作者
达妮娅·杜尔纳斯
发布于：
2026年2月20日凌晨3:48
  • 目录
    What is slopsquatting?
    How is slopsquatting different from typosquatting?
    Is slopsquatting happening now?
    How to protect against slopsquatting attacks
    Conclusion

在npm中，通过注册热门软件包的拼写错误版本，然后等待开发者不小心输入并安装错误软件包的“错拼攻击”已存在十年之久。这并非什么新鲜事——注册表对此已有相应的防护措施。

随后，人工智能登场，再次彻底改变了这一切。Slopsquatting成了新潮流——一种基于人工智能的错别字仿冒手法。与以往押注人类打字错误不同，攻击者如今押注于人工智能的“幻觉”：大语言模型自信推荐的那些根本不存在的软件包名称。有一段时间，这曾被视为一种理论上的风险，但如今已不再是了。人工智能的幻觉正与我们如影随形，而那些“蹭住者”也早已悄然登场。

在本文中，我们将探讨斜坡滑行的原理、研究人员目前在野外发现的情况，以及你实际上可以采取哪些措施来应对这个问题。

什么是斜坡蹲？

“Slopsquatting”，也称为“幻觉蹲”，是指攻击者注册一个人工智能模型容易产生幻觉的软件包名称，然后等待开发者在人工智能的推荐下安装该软件包。蹲点攻击者正是利用了人工智能助手自信地推荐不存在的软件包名称，而开发者又轻信这些推荐，以为它们是自己真正请求的合法软件包。

当开发者使用此软件包名称运行安装时，他们将获得攻击者的软件包。该软件包通常会运行一个安装后脚本，窃取环境中存储的所有凭据（API密钥、云令牌、npm身份验证令牌），并将其转发给攻击者。

虽然有些软件包只是在安装后脚本中包含攻击代码，但更高级的软件包则完全不将恶意代码打包进去，而是利用 npm 对基于 URL 的依赖项的支持，在安装时从外部服务器获取有效载荷。该软件包对任何天真的静态扫描器而言都显得干净，因为其中没有明显的恶意代码。

假设你让AI安装一个JavaScript代码检查工具包。它会提供unused-imports 并询问你是否要安装（如果它不直接跳过询问步骤的话）。这听起来像是你之前用过的那个工具包，于是你便安装了它。

Claude wants to run: npm install unused-imports
[y] Accept  [n] Reject  [e] Edit  [Esc] Cancel

然而，真正的软件包是。结果发现，它是一个恶意软件包，而且令人惊讶的是！你刚刚安装了恶意软件！（顺便一提，这确实是一个真实的恶意软件包，很可能是一次蓄意的仿冒攻击。你可以在Aikido Intel上查一下。 npm 目前已将其置于安全暂停状态。

实际上，大型语言模型经常会“幻觉”出一些包名。Aikido Security 的开发者布道师麦肯齐·杰克逊在 Secure Disclosure 播客的一期节目中描述了他遇到的一次幻觉现象。他请人工智能帮忙将他的 Node.js 项目连接到 OrientDB 数据库——这种技术上可行但并不常见的组合，且没有明显的现成包解决方案。与其承认自己束手无策，这个模型干脆编造了软件包名称。麦肯齐谈到人工智能的思考过程时说：“我确实没什么可给你的，但总得给你点什么，所以我觉得这些软件包可以叫作……”

斜坡抢注与拼错抢注有何不同？

域名仿冒是指攻击者注册一个恶意软件包，其名称与热门软件包非常相似，以至于用户一不小心就会拼错而误入其中。攻击者会寻找一个下载量高、名称容易拼错的软件包，然后在名称中加个连字符、调换两个字母、多加一个字符，抢先于他人“占领”这个拼错的软件包。域名仿冒自2017年起便一直是npm平台上的常见问题，当时一名攻击者发布了一个与热门crossenv包名极为相似的仿冒包。cross-env如今，npm已采取措施加以防范，禁止注册与现有包名过于相似的名称。

拼写错误域名劫持其实是一种错字域名劫持，只不过攻击者押注的不再是人类粗心大意的手指，而是人工智能自信满满地给出错误答案。（没错，目前人工智能的可靠性就是这个水平。）拼写错误域名劫持与我们过去所熟知的错字域名劫持的主要区别在于：前者产生的变体看起来完全不同，而且攻击者可选择的域名数量也更多。

例如，8.7% 的 AI 生成的 Python 包名竟然是有效的JavaScript包。在这种情况下，模型正确地将这些名称与真实事物关联起来，只是所处的生态系统错了。这为攻击者注册其他生态系统的包名提供了绝佳的土壤。

USENIX Security 2025 的研究人员测试了16种模型，涵盖57.6万个代码样本，发现幻觉行为遵循可预测的模式：其中38%是混杂现象，例如express-mongoose，即模型将两种真实事物混为一谈；13%是拼写错误变体；51%则是纯属捏造。可注册的域名池比拼写错误域名攻击提供的要大得多，而且与拼写错误域名攻击不同，这些新域名在npm的保护体系中没有任何东西会被视为‘相似’。

现在正在发生蹲便吗？

我们认为是这样的。我们发现了一些恶意软件包，其名称与抢注域名的模式一致，但我们无法证实攻击者使用这些名称的真实意图。以unused-imports为例，这是我们之前提到过的已确认的恶意软件包。截至2月初，该软件包每周仍能获得233次下载。这些开发者要么正在遵循仍指向此名称的AI建议，要么在他们的依赖树中某个地方引用了该名称并正在重新安装，要么是在未更新的文档或Stack Overflow中发现了它。

然而，研究人员确实在现实生活中发现了并证实了“懒人蹲”的最初迹象。2024年初，Lasso Security的巴尔·拉尼亚多注意到，人工智能模型反复“幻觉”出一个名为的Python软件包。实际工具的安装方式与模型所建议的截然不同，但模型却始终推荐那个更短、根本不存在的版本。Lanyado 以该名称向 PyPI 上传了一个空软件包，以查看会发生什么。

huggingface-cli 三个月内获得了超过3万次真实下载。阿里巴巴竟将这个虚构的安装命令复制粘贴到了其一个公共仓库的README文件中。该软件包本身并无危害，但Lanyado证明了这一策略确实有效。只是幸运的是，Lanyado在攻击者之前发现了这一漏洞。

<b一种自行扩散的AI包幻觉

艾基多的安全研究员查理·埃里克森发现了一件更加离奇的事——一个被幻觉般生成的软件包名称正通过真实的AI基础设施传播，而且还有真实代理试图执行它，而这一切并非有人刻意植入。2026年1月，查理声称这个名为react-codeshift.。这个软件包并不存在，没有作者，也从未注册过。这个名字纯粹是一种典型的jscodeshift混淆幻觉。存在两个名称相似的真实软件包，eact-codemod以及一个由大型语言模型拼凑而成的名称react-codeshift. 

该软件包首次出现在一次提交中，包含47项由大语言模型生成的智能体技能。我们可以推测，有人要求人工智能生成一组编码智能体指令，而在生成过程中，它凭空捏造了一些本无需执行的任务所需的软件包名称。没有人对输出进行过人工审核（至少没有进行过测试），因此这种人工智能幻觉通过GitHub被永久保存了下来。

当查理在进行无人认领包裹的调查时发现了它，这个根本不存在的包裹的名字已通过分叉传播至237个仓库，并被翻译成了日语。在查理认领之后，该包裹每天仍会获得几笔下载量。这些是遵循技能指令并在真实环境中触发 npm 安装的人工智能代理。如果攻击者率先注册了它，就可能引发更大规模的恶意软件投递攻击，且这种攻击会自动扩散

如何防范冲浪攻击

请核实发布者，而不仅仅是名称

显而易见的答案是在安装前验证软件包名称，但实际上这并不那么简单。下载量并不是一个可靠的信号（我们发现，恶意软件包依然会有正常的每日下载量）。真正重要的是发布者：是谁注册了这个软件包、何时注册的，以及这是否符合你对合法维护者的预期？ 一个声称是插件的软件包，却没有维护者信息，且注册日期为上周二，无论其下载量如何，都应引起警惕。

将自动软件包安装视为特权操作

如果您运行的AI智能体能够无需确认即可安装软件包、Claude Code处于绕过模式、采用智能体编码设置，或CI流水线拥有广泛的npm权限，那么您通常作为人类所执行的验证步骤将不复存在。只要智能体拥有相应权限，它便会直接继续执行。这正是slopsquatting所基于的威胁模型，因此请据此设定相应的权限范围。

扫描您的完整依赖树

一些被误报的软件包名称最终变成了嵌套依赖，而非直接安装，这意味着它们不会出现在你的 package.json 中。软件成分分析（SCA）扫描器会查看你的完整依赖树，以发现隐藏的、深藏不露的恶意软件包。

使用 SafeChain 实现 npm 级别保护

Aikido SafeChain是一个开源封装工具，用于npm, npx, yarn和pnpm,它会在软件包安装命令到达您的设备之前拦截这些命令，并将其与Aikido Intel进行比对。

结论

未认领的软件包名称一直都可以被认领，但现在我们有了能够自信地为我们提供虚假软件包名称以供安装的人工智能，以及能在各个仓库中传播这些名称的人工智能代理。

随着氛围编程成为常态，越来越多以龙虾为主题的AI智能体开始在无人干预的情况下进行编码（请阅读我们关于“为何试图保护OpenClaw是荒谬之举”的文章），人类在恶意软件包名称真正运行之前将其揪出的机会窗口正日益缩小。我们发现，大语言模型编造的名称具有一致性和可重复性，攻击者也已开始利用这一点。

检查您的依赖树。验证发布者。并使用一种介于您的包管理器与注册表之间的工具，让它为您完成检查工作。
