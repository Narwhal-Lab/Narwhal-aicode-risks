19个NPM包裹在重大供应链攻击中被攻破

阅读8分钟

伟大的npm妥协：事后分析

今天发现了一起针对NPM生态系统的协调供应链攻击，这一迅速且广泛的攻破行动毫无疑问地表明了开源威胁日益令人担忧的性质。虽然许多头条新闻聚焦于一封钓鱼邮件，但真正的故事在于随后出现的复杂多层次载荷，旨在悄无声息地从加密钱包中窃取数据。

这不是一次随意的抢夺行动，而是高度针对性的行动，利用社会工程技术植入了先进的客户方金融武器。这完美地展示了为什么传统的被动式安全方法已经不再足够。

这种新的威胁层级要求采取新的安全策略。仅仅寻找问题已不够;我们需要一个安全模型，能够通过理解代码和生态系统的微妙上下文来预判威胁。

但首先，先说攻击。

npm供应链攻击：社会工程大师课

这次攻击是由针对广泛使用NPM软件包维护者的钓鱼攻击发起的客户端攻破。攻击者控制了维护者的账户，并利用该权限向19个热门依赖系统注入恶意代码，这些依赖累计每周下载量超过20亿次。注入的有效载荷在所有受影响的包中均为相同，旨在悄无声息地从与加密资产交互的系统中抽取敏感信息。

被攻破的软件包及其恶意版本包括：

  • ANSI-styles@6.2.2
  • debug@4.4.2
  • chalk@5.6.1
  • supports-color@10.2.1
  • 条ansi@7.1.1
  • ANSI-regex@6.2.1
  • Wrap-ansi@9.0.1
  • color-convert@3.1.1
  • color-name@2.0.1
  • IS-arrayish@0.3.3
  • 切片ansi@7.1.1
  • color@5.0.1
  • color-string@2.1.1
  • simple-swizzle@0.2.3
  • 支持-hyperlinks@4.1.1
  • has-ansi@6.0.1
  • 粉笔-template@1.1.1
  • backslash@0.2.1
  • 错误-ex@1.3.3

妥协的经过

这种漏洞可能以两种关键方式使组织面临风险：

  1. 其中一个被攻破的软件包首次被使用，恶意版本最终进入了该环境中。
  2. 一个集装箱被重建，过程中还包括其中一个易受攻击的包裹。由于没有使用文件锁来钉顶包版本，构建会从npm中拉取最新和被攻破的版本，这些版本随后嵌入到工件中。
  3. 如果执行 npm 更新命令，执行时会将package.json中的版本提升到最新版本（且已被攻破）。

此时重建工件应能从npm中恢复修复包版本，有效修复漏洞。

攻击向量与有效载荷的技术分析

最初的攻破始于一封来自虚假域名npmjs.help的复杂钓鱼邮件，旨在冒充官方npm注册（npmjs.com）。这封邮件利用了一贯可靠的社交工程技术，诱使维护者在一个假登录页面上“更新”他们的2FA凭证。凭证和令牌随后被泄露到 websocket-api2.publicvm.com 的攻击者控制的端点。

注入包中的恶意代码充当了“Web3 抽干器”，即浏览器内人攻击，旨在劫持加密货币活动。一旦激活，它会静默监控连接的钱包，并通过交易交换等技术操控交易。

该载荷的主要功能是通过连接标准浏览器API（如fetch和XMLHttpRequest），以及Web3 API（如window.ethereum）和钱包如MetaMask和Phantom使用的API，拦截和操作加密货币交易。由于代码会篡改网页上显示的数据，甚至存款字段或二维码也可能在用户不知情的情况下被更改，导致资金、代币和批准直接流向攻击者。

该恶意软件能够对多种加密货币进行地址交换，包括以太坊、比特币、Solana、Tron、莱特币和比特币现金。它还针对链上智能合约功能，如ERC-20代币转移和审批，窃取了各种资产。地址交换逻辑尤其先进，使用字符串相似度算法，如Levenshtein距离，找到与合法地址“最接近”的攻击者地址，使欺诈地址几乎与目标地址完全相同。

OX安全行动

此次攻击凸显了对新型分层应用安全方法的关键需求。OX安全平台采用AI驱动的整体安全方法，以适应AI驱动的世界。

OX的实时威胁情报和SBOM关联引擎能够识别恶意版本，并在受影响环境中自动触发警报。软件物料清单（SBOM）是构成软件应用的所有组件的完整清单，提供第三方和开源依赖的集中记录。这使得我们的引擎能够快速识别哪些环境受到漏洞或威胁的影响。

Conclusion

For teams without access to automated security solutions, the immediate priority is to audit projects manually. Remove or downgrade the compromised package versions, rebuild applications from clean sources, and review deployed environments for any injected code. Developers and end-users should revoke existing token approvals, rotate secrets, and, when possible, move funds to fresh wallets created offline.

The fragility of trust in open-source ecosystems is now clear. OX Security’s layered approach, combining SBOM visibility, runtime monitoring, and threat intelligence ensures that even deeply nested threats are surfaced and neutralized before damage is done. 

We’ll continue to monitor the npm ecosystem and provide updates as the situation evolves. For real-time alerts and remediation guidance, visit your OX Security dashboard or contact your account team.
